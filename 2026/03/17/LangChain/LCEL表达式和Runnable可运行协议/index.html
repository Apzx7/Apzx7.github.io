LCEL表达式和Runnable可运行协议

AI LangChain LCEL Runnable

Runnable与LCEL：让AI应用开发像“搭积木”一样简单

在构建基于大语言模型（LLM）的AI应用时，你是否遇到过这样的困境：每个组件（提示词、模型、解析器）的调用方式都不一样，组合它们需要写大量“胶水代码”，想加个新功能就得重写整个流程？Runnable接口和LCEL表达式语言正是LangChain为解决这些问题而设计的“神器组合”。它们将复杂的AI应用开发，变成了像搭乐高积木一样直观、高效的体验。

一、核心概念：用“说人话”的方式理解

1. Runnable：所有组件的“通用USB接口”

想象一下，你的手机、键盘、U盘都有不同的充电口，每次连接都要找对应的线，多麻烦？在早期的LangChain中，提示模板、语言模型、输出解析器等组件就像这些设备，各有各的调用方法（有的用.format()，有的用.generate()，有的用.parse()）。

Runnable就是LangChain为所有组件制定的 “标准通信协议” 。它定义了一组统一的调用方法（如invoke、stream、batch），让不同的组件能够用同一种方式“对话”。只要一个组件实现了Runnable接口，它就能和其他Runnable组件无缝协作。

2. LCEL：积木的“拼接说明书”

有了标准接口的积木，怎么把它们又快又好地搭起来呢？这就是LCEL（LangChain Expression Language） 的用武之地。

LCEL是一种声明式的领域特定语言，核心就是用管道符 | 来连接组件。你不需要详细描述“第一步怎么做，第二步怎么做”（命令式编程），只需要声明“数据应该按这个顺序流经这些组件”（声明式编程）。框架会自动帮你处理中间的所有细节。

一个最经典的LCEL链看起来就像这样：

chain = prompt | model | output_parser

它的含义一目了然：数据先流向prompt模板格式化，然后结果自动传给model模型调用，最后模型的输出交给output_parser解析。

二、解决了什么问题？告别“胶水代码地狱”

在Runnable和LCEL出现之前，LangChain的开发体验存在几个核心痛点，而它们完美地解决了这些问题：

  1. 组件调用不统一，适配代码繁琐

    • 传统方式：你需要记住每个组件的特定方法，并手动传递数据。
    # 传统方式：各组件调用方法五花八门[9](@ref)
prompt_text = prompt.format(topic="猫")  # 提示词用 .format()
model_out = model.generate(prompt_text) # 模型用 .generate()
result = parser.parse(model_out)        # 解析器用 .parse()
    • LCEL方式：所有组件统一用.invoke()，用|连接，无需手动处理中间结果。
    chain = prompt | model | parser
result = chain.invoke({"topic": "猫"}) # 一个方法搞定所有[9](@ref)

  2. 生产级功能需要手动实现
    流式输出、批量处理、异步支持这些提升体验和性能的功能，在传统方式下需要开发者自己实现。而任何用LCEL构建的链都原生支持这些特性，无需额外代码。

    • chain.stream()：用于流式输出，提升用户体验。
    • chain.batch()：用于批量处理，自动优化并行执行。
    • chain.ainvoke()：用于异步调用，适配高并发服务。

  3. 代码冗长，逻辑难以追踪
    当应用变得复杂，需要串联多个步骤时，传统面向对象的方式会让代码迅速膨胀，逻辑流隐藏在多个类的实例化和调用中。LCEL的管道语法让数据流清晰可见，整个工作流程一目了然。

  4. 定制和扩展困难
    如果想在链中加入一个自定义处理步骤，传统方式可能需要创建一个新的子类并重写方法，非常笨重。在LCEL范式下，任何实现了Runnable接口的自定义函数，都可以直接用|插入到链中，扩展性极强。

三、基础代码示例：从“传统”到“现代”的直观对比

让我们通过一个具体任务来感受两者的差异：让AI生成一个关于某主题的笑话。

传统方式（已不推荐，仅作对比）

from langchain.llms import OpenAI
from langchain.prompts import PromptTemplate
from langchain.chains import LLMChain

# 1. 实例化模型
llm = OpenAI(temperature=0.7)
# 2. 构建提示模板
prompt = PromptTemplate(
    input_variables=["product"],
    template="What is a good name for a company that makes {product}?",
)
# 3. 构建链，将零件通过参数传入
chain = LLMChain(llm=llm, prompt=prompt)
# 4. 显式调用run()
result = chain.run("colorful socks")
print(result)

这段代码步骤繁多，且LLMChain在最新版本中已被弃用。

现代方式（使用LCEL）

from langchain_core.prompts import ChatPromptTemplate
from langchain_openai import ChatOpenAI
from langchain_core.output_parsers import StrOutputParser

# 1. 定义提示模板
prompt = ChatPromptTemplate.from_template("讲一个关于{topic}的笑话")
# 2. 初始化模型
model = ChatOpenAI(model="gpt-4o-mini", temperature=0.7)
# 3. 定义输出解析器（提取纯文本）
parser = StrOutputParser()

# 4. 使用管道符 | 声明式组合链（核心！）
chain = prompt | model | parser

# 5. 使用统一的invoke方法执行
result = chain.invoke({"topic": "程序员"})
print(result)

# 你还可以轻松尝试其他强大功能
# 流式输出：for chunk in chain.stream({"topic": "程序员"}): print(chunk, end="")
# 批量处理：results = chain.batch([{"topic": "猫"}, {"topic": "狗"}])

代码简洁，逻辑清晰，且自动获得了流式、批量等能力。

四、LCEL不是万能的：选择合适的工具

尽管LCEL非常强大，但它主要是一种编排解决方案，擅长优化链的运行时执行。LangChain官方给出了清晰的选用指南：

  • 简单调用：如果只是单次调用LLM，直接使用模型即可，无需LCEL。
  • 简单链：对于“提示+模型+解析器”或简单检索这类线性流程，LCEL是最佳选择，能最大化利用其优势。
  • 复杂工作流：当应用需要复杂的状态管理、循环、条件分支或多个智能体协作时（例如一个需要反复查询、决策的客服机器人），应该使用LangGraph。LangGraph允许你以图结构定义流程，更适合复杂逻辑。值得注意的是，你可以在LangGraph的单个节点内部使用LCEL，实现优势互补。

小结

总而言之，Runnable接口通过统一调用标准，解决了组件间“语言不通”的问题；LCEL表达式通过声明式的管道语法，解决了工作流“搭建繁琐”的问题。它们共同将开发者从“胶水代码地狱”中解放出来，让开发者能更专注于业务逻辑本身，像搭积木一样快速、优雅地构建出功能强大且易于维护的AI应用。

对于大多数线性任务，从今天开始就使用 prompt | model | parser 这样的LCEL链吧，这是现代LangChain开发的起点和最佳实践。

LangChain 的 Runnable 接口 是其现代架构（0.1+版本）的核心，它通过定义一套统一的调用标准（如 invoke, stream, batch），将提示模板、语言模型、输出解析器等异构组件“标准化”，使它们能够像乐高积木一样被无缝组合和调用。其实际应用贯穿于构建高效、可维护的LLM应用工作流的各个环节。

以下是 Runnable 接口及其具体实现（原语）的核心应用场景：

1. 构建标准化、可组合的链式工作流（核心应用）

这是 Runnable 最基础也是最广泛的应用。通过 LCEL（LangChain Expression Language） 的管道操作符 |，开发者可以声明式地将多个 Runnable 组件串联起来，形成一个清晰的数据流。

  • 应用场景：任何需要多步骤处理的任务，例如“用户输入 -> 提示词格式化 -> 模型调用 -> 输出解析”。

  • 代码示例：

    from langchain_core.prompts import ChatPromptTemplate
from langchain_openai import ChatOpenAI
from langchain_core.output_parsers import StrOutputParser

prompt = ChatPromptTemplate.from_template("请将{text}翻译成英文。")
model = ChatOpenAI(model="gpt-4o-mini")
parser = StrOutputParser()

# 使用 | 构建链，所有组件都实现了Runnable接口
chain = prompt | model | parser
result = chain.invoke({"text": "你好，世界"})

    这种方式代码简洁，逻辑一目了然，并且自动获得了流式、批量、异步等生产级能力。

2. 数据传递、预处理与分支路由

对于复杂工作流，经常需要操作数据流，例如传递原始输入、根据条件执行不同分支。

  • RunnablePassthrough：传递与扩充数据

    • 应用场景：在链中需要保留原始输入或为后续步骤添加额外数据时使用。常与 RunnableLambda 配合，在不阻断主数据流的情况下注入信息。

    • 代码示例：保留用户问题，并为其添加一个处理时间戳。

      from langchain_core.runnables import RunnablePassthrough, RunnableLambda
from datetime import datetime

chain = RunnablePassthrough.assign(
    timestamp=RunnableLambda(lambda _: datetime.now().isoformat())
)
# 输入: {"question": "今天天气如何？"}
# 输出: {"question": "今天天气如何？", "timestamp": "2026-03-14T10:00:00"}

  • RunnableBranch：实现条件路由

    • 应用场景：根据输入内容动态选择不同的处理子链。例如，构建一个客服系统，将“技术问题”、“账单问题”和“其他问题”路由给不同的专业链或提示词进行处理。

    • 代码示例：根据问题关键词路由到不同的回答链。

      from langchain_core.runnables import RunnableBranch

route_chain = RunnableBranch(
    (lambda x: "价格" in x["question"], price_chain),
    (lambda x: "故障" in x["question"], tech_support_chain),
    default_chain # 默认处理其他问题
)

3. 并行执行与结果聚合

为了提升应用性能，经常需要并行执行多个独立任务。

  • RunnableParallel (或 RunnableMap)：并行执行

    • 应用场景：需要同时获取多种信息或执行多个独立推理步骤时。例如，在RAG系统中，并行检索相关文档和分析用户问题的情感；或者为一个主题同时生成摘要和关键词列表。

    • 代码示例：并行获取某个城市的景点和美食推荐。

      from langchain_core.runnables import RunnableParallel

parallel_chain = RunnableParallel({
    "attractions": attractions_chain, # 一个生成景点的链
    "food": food_chain # 一个生成美食的链
})
result = parallel_chain.invoke({"city": "南京"})
# 输出: {"attractions": ["中山陵", ...], "food": ["盐水鸭", ...]}

4. 集成自定义逻辑与外部系统

LLM应用常需与现有业务逻辑或外部API交互，Runnable为此提供了优雅的集成方式。

  • RunnableLambda：封装任意函数

    • 应用场景：这是连接LangChain与外部世界的“万能适配器”。可用于数据清洗、调用内部API、格式化输出、插入日志记录、敏感词过滤等任何自定义操作。

    • 代码示例：在模型输出后，调用一个外部翻译API进行后续处理。

      from langchain_core.runnables import RunnableLambda
def my_translation_func(text: str) -> str:
    # 调用某个内部翻译服务
    return call_internal_translation_api(text, target_lang="ja")

custom_chain = some_llm_chain | RunnableLambda(my_translation_func)

5. 管理有状态的对话（记忆）

构建多轮对话机器人时，管理对话历史是关键。

  • RunnableWithMessageHistory：管理对话记忆

    • 应用场景：为任何链添加对话历史管理能力，构建能进行连贯多轮对话的聊天机器人、客服助手等。

    • 关键点：在现代Runnable体系中，记忆是显式管理的。需要在提示模板中预留 {history} 占位符，并通过 RunnableWithMessageHistory 将链与一个历史存储函数绑定。

    • 代码示例（简化）：

      from langchain_core.runnables.history import RunnableWithMessageHistory

chat_chain = RunnableWithMessageHistory(
    base_chain, # 你的基础问答链
    get_session_history, # 一个返回历史存储对象的函数
    input_messages_key="question",
    history_messages_key="history"
)
# 调用时需传入session_id以区分不同对话
chat_chain.invoke({"question": "我上一条问了什么？"}, config={"configurable": {"session_id": "user123"}})

总结

总而言之，Runnable接口的实际应用远不止于简单的链式调用。它通过 RunnablePassthrough、RunnableParallel、RunnableBranch、RunnableLambda 等一组强大的“编排原语”，赋能开发者以声明式的方式构建从简单的线性问答到包含并行处理、条件路由、状态管理和外部集成的复杂AI工作流。这种设计极大地提升了代码的模块化、可读性和可维护性，是LangChain支撑生产级LLM应用开发的基石。
